# ADR-019: Generated Code Package Layout

## Status

Accepted

## Date

2026-03-01

## Context

The generated code must be organized clearly so that users can:

1. Find types, endpoints, and auth helpers easily
2. Regenerate without overwriting user code (generated code lives in a separate package/directory from user code)
3. Import only the generated package they need (one package per API, not a monolithic library bundling all possible APIs)
4. Understand the structure at a glance

### Design constraints

- Generated files must be clearly marked (header comment + filename convention)
- No generated code should be manually edited (regeneration would overwrite)
- The runtime library (`openapigo`) is a separate module from generated code
- Generated code imports the runtime library, never vice versa

## Decision

### Two Packages: Runtime Library + Generated Code

```
github.com/mkusaka/openapigo/       ← Runtime library (this repo)
├── client.go                        ← Client, Do(), DoRaw(), DoStream()
├── endpoint.go                      ← Endpoint[Req, Resp] type
├── middleware.go                     ← Middleware interface
├── nullable.go                      ← Nullable[T] type
├── errors.go                        ← APIError, ValidationError, etc.
├── serializer.go                    ← Parameter serialization
├── file.go                          ← File type for uploads
├── date.go                          ← Date type
├── helpers.go                       ← isUnique, hasAll, marshalMerge, etc.
└── codec.go                         ← JSONCodec interface

user-project/
├── api/                             ← Generated package (user configurable)
│   ├── types.go                     ← Schema types (structs, enums, aliases)
│   ├── operations.go                ← Request/response types per operation
│   ├── endpoints.go                 ← Endpoint variables
│   ├── auth.go                      ← Security scheme middleware constructors
│   ├── validators.go                ← Validate() methods (optional)
│   └── iterators.go                 ← Pagination iterators (optional)
└── main.go                          ← User code
```

### Generated File Convention

Every generated file starts with a header comment:

```go
// Code generated by openapigo. DO NOT EDIT.
//
// Source: petstore.yaml
// Generator version: 0.1.0
```

Go tooling (e.g., `go generate`, IDE file icons) recognizes the `// Code generated ... DO NOT EDIT.` comment.

### File Breakdown

#### types.go — Schema Types

Contains all types generated from `components/schemas`:

```go
// Code generated by openapigo. DO NOT EDIT.
package api

import (
    "time"
    "github.com/mkusaka/openapigo"
)

type Pet struct {
    ID   string `json:"id"`
    Name string `json:"name"`
    Tag  *string `json:"tag,omitzero"`
}

type PetStatus string

const (
    PetStatusActive   PetStatus = "active"
    PetStatusInactive PetStatus = "inactive"
)

func PetStatusValues() []PetStatus { /* ... */ }
func (v PetStatus) Validate() error { /* ... */ }

// Request/Response type variants (ADR-007)
type UserRequest struct { /* excludes readOnly */ }
type UserResponse struct { /* excludes writeOnly */ }

// Composition types (ADR-002)
type CatOrDogOneOf struct {
    Cat *Cat
    Dog *Dog
}
// UnmarshalJSON, MarshalJSON, IsCat, IsDog, etc.
```

#### operations.go — Request/Response Types Per Operation

Contains request structs and per-operation error types:

```go
// Code generated by openapigo. DO NOT EDIT.
package api

// GetPet
type GetPetRequest struct {
    PetID string `path:"petId"`
}

type GetPet404Error struct {
    openapigo.APIError
    Payload NotFoundError
}

// ListPets
type ListPetsRequest struct {
    Limit  *int    `query:"limit,omitzero"`
    Cursor *string `query:"cursor,omitzero"`
}

type ListPetsResponse struct {
    Items      []Pet   `json:"items"`
    NextCursor *string `json:"next_cursor,omitzero"`
}

// CreatePet
type CreatePetRequest struct {
    Body CreatePetBody `body:"application/json"`
}

type CreatePetBody struct {
    Name string  `json:"name"`
    Tag  *string `json:"tag,omitzero"`
}
```

#### endpoints.go — Endpoint Variables

Contains all Endpoint variable declarations:

```go
// Code generated by openapigo. DO NOT EDIT.
package api

import "github.com/mkusaka/openapigo"

// GetPet retrieves a pet by ID.
//
// GET /pets/{petId}
// Security: bearerAuth (http/bearer)
var GetPet = openapigo.Endpoint[GetPetRequest, Pet]{
    Method: "GET",
    Path:   "/pets/{petId}",
}.WithErrors(
    openapigo.ErrorHandler{Status: 404, Parse: parseGetPet404Error},
)

// ListPets lists all pets.
//
// GET /pets
// Security: none
var ListPets = openapigo.Endpoint[ListPetsRequest, ListPetsResponse]{
    Method: "GET",
    Path:   "/pets",
}

// CreatePet creates a new pet.
//
// POST /pets
// Security: bearerAuth (http/bearer)
var CreatePet = openapigo.Endpoint[CreatePetRequest, Pet]{
    Method: "POST",
    Path:   "/pets",
}
```

#### auth.go — Security Scheme Middleware

Contains generated auth helpers (per ADR-018):

```go
// Code generated by openapigo. DO NOT EDIT.
package api

func WithBearerAuth(token string) openapigo.Middleware { /* ... */ }
func WithBearerAuthFunc(tokenFunc func(ctx context.Context) (string, error)) openapigo.Middleware { /* ... */ }
func WithAPIKeyAuth(apiKey string) openapigo.Middleware { /* ... */ }
```

#### validators.go — Validate() Methods (Optional)

Generated only when `--skip-validation` is NOT set:

```go
// Code generated by openapigo. DO NOT EDIT.
package api

func (v Pet) Validate() error { /* ... */ }
func (v CreatePetBody) Validate() error { /* ... */ }
```

#### iterators.go — Pagination Iterators (Optional)

Generated only when pagination patterns are detected:

```go
// Code generated by openapigo. DO NOT EDIT.
package api

func ListPetsIter(ctx context.Context, client *openapigo.Client, req ListPetsRequest) iter.Seq2[Pet, error] {
    /* ... */
}
```

### Large API Splitting

For very large APIs (100+ schemas, 50+ operations), the generator can split files by tag:

```
api/
├── types.go              ← Shared/untagged schemas
├── types_pets.go          ← Schemas used by "pets" tag
├── types_users.go         ← Schemas used by "users" tag
├── operations_pets.go     ← Pet operation request/response types
├── operations_users.go    ← User operation request/response types
├── endpoints.go           ← All endpoints (single file is fine)
├── auth.go
└── validators.go
```

Enabled via CLI flag `--split-by-tag`. Default is single-file-per-category.

**Schema placement rules for `--split-by-tag`**: When a schema is referenced by operations in **multiple tags**, it is placed in the shared `types.go` file (not duplicated across tag files). The generator tracks which schemas are tag-exclusive vs shared during analysis:

| Schema usage | Placement |
|---|---|
| Referenced by operations of exactly one tag | `types_{tag}.go` |
| Referenced by operations of multiple tags | `types.go` (shared) |
| Not referenced by any operation (orphan) | `types.go` (shared) |
| Inline schema (not in components) | Same file as the operation |

**Tag name sanitization**: Tag names from the OpenAPI spec are sanitized for use as file name suffixes: lowercase, hyphens/spaces/special characters replaced with underscores, leading digits prefixed with underscore. **Go build constraint avoidance**: Sanitized tag suffixes are checked against Go's special filename conventions. Tags that would produce suffixes matching Go OS names (`linux`, `darwin`, `windows`, `freebsd`, etc.), architecture names (`amd64`, `arm64`, `386`, `arm`, etc.), or `test` are given a `_gen` **suffix** to avoid unintended build constraints or test-file exclusion. Example: tag `test` → `types_test_gen.go` (not `types_test.go`), tag `linux` → `types_linux_gen.go` (not `types_linux.go`). **Note**: A `tag_` **prefix** (e.g., `types_tag_linux.go`) does NOT work because Go's build constraint matching checks the substring after the last `_` in the filename — `tag_linux` still ends with `_linux`, triggering the GOOS constraint. The `_gen` suffix approach (`types_linux_gen.go`) works because the last `_`-separated segment is `gen`, which is not a recognized OS, architecture, or `test`. If two tags produce the same sanitized suffix, the generator emits an error: `ERROR: tags "foo-bar" and "foo bar" both sanitize to "foo_bar". Rename one tag or use --split-by-tag=false.` **Post-sanitization collision detection**: The generator also checks sanitized tag suffixes against the base generated filenames (`types`, `operations`, `endpoints`, `auth`, `validators`, `iterators`). A tag that sanitizes to any of these would produce filenames like `types_types.go` — while not a name collision per se, this is confusing. The generator emits a warning: `WARN: tag "types" produces confusing filename "types_types.go". Consider renaming the tag.`

**Operations with multiple tags**: When an operation has multiple tags, its request/response types are placed in the file of the **first** tag. The endpoint variable references the types regardless of file placement (all files are in the same Go package).

**Operations with no tag**: Operations without a tag are placed in the base files (`operations.go`, `types.go`).

### CLI Interface

```bash
# Basic usage
openapigo generate -i petstore.yaml -o ./api

# With options
openapigo generate \
  -i petstore.yaml \
  -o ./api \
  --package api \
  --skip-validation \
  --split-by-tag \
  --method-client \
  --uuid-package=github.com/google/uuid \
  --no-read-write-types
```

| Flag | Default | Description |
|------|---------|-------------|
| `-i, --input` | (required) | Path or URL to OpenAPI spec |
| `-o, --output` | `./api` | Output directory |
| `--package` | directory name | Go package name (sanitized — see below) |
| `--skip-validation` | `false` | Don't generate `Validate()` methods |
| `--split-by-tag` | `false` | Split files by OpenAPI tag |
| `--method-client` | `false` | Generate method-based client wrapper |
| `--uuid-package` | (none) | Use external UUID package |
| `--no-read-write-types` | `false` | Don't split Request/Response types |
| `--strict-enums` | `false` | Reject unknown enum values on unmarshal |
| `--additional-properties` | `ignore` | How to handle absent `additionalProperties`: `ignore` (plain struct), `preserve` (capture as `json.RawMessage` map), `strict` (treat absent as `true`, full map generation). See ADR-006. |
| `--validate-on-unmarshal` | `false` | Call `Validate()` after unmarshal. See ADR-013. |
| `--validate-on-unmarshal-skip` | (none) | Comma-separated categories to skip in auto-validation (`enum`, `const`, etc.). See ADR-013. |
| `--format-mapping` | (none) | Custom format → Go type mappings |

### go generate Integration

Users add a `go:generate` directive:

```go
//go:generate openapigo generate -i ../petstore.yaml -o . --package api
package api
```

Or use Go 1.24's tool directive:

```
// go.mod
tool github.com/mkusaka/openapigo/cmd/openapigo
```

```bash
go tool openapigo generate -i petstore.yaml -o ./api
```

### What Is NOT Generated

- `go.mod` / `go.sum` — users manage their own module
- Test files — generated code is deterministic, testing the generator is sufficient
- Documentation files — GoDoc comments are inline
- Configuration files — CLI flags only

## Consequences

### Positive

- **Clear separation**: types, operations, endpoints, auth in separate files
- **Regeneration-safe**: `DO NOT EDIT` header + no mixed generated/custom files
- **Minimal file count**: 4-6 files for most APIs, not one-per-schema
- **Standard Go tooling**: `go:generate`, `go tool`, standard package import
- **Large API support**: `--split-by-tag` for APIs with many operations

### Negative

- **Single package**: all generated code in one Go package. Very large APIs may want sub-packages, but Go's import cycle rules make this complex with shared types.
- **Regeneration overwrites everything in the generated package**: users cannot add methods to generated types within the generated files. They must define extensions in separate files outside the generated directory (e.g., wrapper types, helper functions). This is the standard Go code generation pattern (cf. protobuf, sqlc).
- **File count grows with flags**: `--method-client` + `--split-by-tag` + validators produces many files

### Risks

- Package name collisions: if the user's project already has a package named `api`, they must use `--package` to choose a different name.
- **Package name sanitization**: Go package names must be valid Go identifiers. By convention (and per `go vet` / `golint` recommendations), package names should be lowercase, without hyphens, dots, or leading digits. While the Go specification allows uppercase letters in package names, the standard toolchain and community convention strongly discourage it. The generator enforces lowercase as part of sanitization. When the output directory name or `--package` value contains invalid or non-idiomatic characters, the generator sanitizes:
  - Hyphens and dots → underscores (`my-api` → `my_api`)
  - Leading digits → prefix with `pkg` (`3rdparty` → `pkg3rdparty`)
  - Uppercase → lowercase
  - Go reserved words → suffix with `_api` (`type` → `type_api`)
  - `main` → suffix with `_api` (`main` → `main_api`). While `go/token.IsIdentifier()` accepts `main`, a `package main` cannot be imported as a library, which defeats the purpose of generated code. Note: `init` is a valid and importable package name (unlike `main`); `func init()` is special but `package init` has no special semantics. The generator does not reject `init` as a package name.
  - Result is `_` (blank identifier), empty, or all underscores → replaced with `api` (the default)
  - The generator validates the final name with `go/token.IsIdentifier()` as a safety net, **after** the above special-case rejections
  - The generator emits a warning when sanitization changes the name: `WARN: package name "my-api" sanitized to "my_api"`
- Very large specs (1000+ schemas) may produce files too large for some editors. The `--split-by-tag` flag mitigates this, but a `--split-by-schema` option may be needed for extreme cases.
