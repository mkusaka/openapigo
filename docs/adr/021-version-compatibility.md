# ADR-021: Runtime/Generator Version Compatibility

## Status

Accepted

## Date

2026-03-02

## Context

The openapigo ecosystem has two independently-versioned artifacts:

1. **Generator CLI** (`openapigo` binary): produces Go source files from an OpenAPI spec
2. **Runtime library** (`github.com/mkusaka/openapigo` Go module): provides `Client`, `Do()`, `Endpoint[Req, Resp]`, `Nullable[T]`, etc.

In production, these two artifacts can become mismatched:

- A user generates code with generator v0.5.0, then updates the runtime to v0.6.0 via `go get` without regenerating
- A user pins the runtime but upgrades the generator, producing code that references new runtime APIs

**protobuf's prior art**: Protocol Buffers explicitly documents a [cross-version runtime guarantee](https://protobuf.dev/support/cross-version-runtime-guarantee/): generated code from protoc-gen-go vX works with the protobuf Go runtime module for a defined range of versions. This is a critical usability contract that openapigo needs to establish.

### Versioning constraints

- The generator and runtime are in the **same Go module** (`github.com/mkusaka/openapigo`). The CLI is at `github.com/mkusaka/openapigo/cmd/openapigo`.
- Go modules use semantic versioning. Pre-1.0 (`v0.x.y`), any minor version bump can break.
- Generated code has a header comment including the generator version (ADR-019).

## Decision

### Single Module, Same Version

The generator CLI and the runtime library are versioned together in a single Go module. A release `v0.5.0` means both the CLI and the runtime are at `v0.5.0`.

This avoids the cross-version matrix problem entirely for the pre-1.0 phase. Users always have a matched pair.

### Compatibility Guarantee

#### Pre-1.0 (`v0.x.y`)

- **Minor version bumps** (`v0.x.0` → `v0.x+1.0`) MAY contain breaking changes to:
  - Generated code shape (struct fields, method signatures)
  - Runtime API (function signatures, type definitions)
  - CLI flags
- **Patch version bumps** (`v0.x.y` → `v0.x.y+1`) contain ONLY:
  - Bug fixes
  - Documentation improvements
  - Performance improvements that don't change behavior
- **Skew tolerance**: Generated code from `v0.x.y` is guaranteed to work with runtime `v0.x.z` (same minor, any patch). Cross-minor compatibility is NOT guaranteed.
- **Regeneration required**: When upgrading to a new minor version, users MUST regenerate their code. The generator emits a warning when it detects stale generated code (see below).

#### Post-1.0 (`v1.x.y`)

Once we reach `v1.0.0`:

- **Major version bumps** (`v1` → `v2`) MAY contain breaking changes. Go module path changes (`/v2` suffix).
- **Minor version bumps** (`v1.x` → `v1.x+1`) are backwards-compatible. Generated code from `v1.x` works with runtime `v1.x+n`.
- **Skew tolerance**: Generated code from `v1.x.y` works with runtime `v1.z.w` where `z >= x`. The runtime is forward-compatible with older generated code within the same major version. Newer generated code (e.g., from `v1.5.0`) does NOT work with older runtimes (e.g., `v1.4.0`) if it uses new runtime APIs — this fails at compile time with missing symbol errors.

### Stale Code Detection

Generated files include the generator version in the header comment (ADR-019):

```go
// Code generated by openapigo. DO NOT EDIT.
//
// Source: petstore.yaml
// Generator version: 0.5.0
```

The runtime library includes its own version as a constant:

```go
// In the runtime library
const Version = "0.5.0"
```

**Build-time detection**: The generator emits a version check in the generated code that produces a **compile-time error** when the runtime version is incompatible:

```go
// Code generated by openapigo. DO NOT EDIT.
//
// Source: petstore.yaml
// Generator version: 0.5.0
package api

import "github.com/mkusaka/openapigo"

// Ensures the generated code is compatible with the runtime library.
// If this fails to compile, run `openapigo generate` to regenerate.
var _ = openapigo.RuntimeCompatV0_5
```

**Header format consistency**: The **first line** of every generated file is always `// Code generated by openapigo. DO NOT EDIT.` — this is the canonical header used by ADR-019's stale file cleanup (exact prefix match on line 1). The generator version is placed on a **separate comment line** (`// Generator version: 0.5.0`), not embedded in the first line. This ensures stale file detection works regardless of generator version.

The runtime library exports compatibility marker constants:

```go
// runtime/compat.go
const RuntimeCompatV0_5 = true
// When v0.6 ships and removes backward compat:
// RuntimeCompatV0_5 is deleted → old generated code fails to compile
// RuntimeCompatV0_6 = true is added
```

This gives a clear compile-time error when versions are mismatched, rather than subtle runtime bugs.

### CLI Version Check

The generator also checks the target project's `go.mod` for the runtime version:

```bash
$ openapigo generate -i petstore.yaml -o ./api
WARN: go.mod requires github.com/mkusaka/openapigo v0.4.0 but this generator is v0.5.0.
      Generated code may require runtime features not present in v0.4.0.
      Run: go get github.com/mkusaka/openapigo@v0.5.0
```

This is a warning, not an error, because the user may be about to run `go get`.

### Changelog and Migration

Each minor version release includes:

1. **CHANGELOG.md**: Lists all changes, clearly marking breaking changes
2. **Migration guide**: For breaking changes, describes what to change and provides a script when possible:
   ```bash
   # Example: v0.5 → v0.6 migration
   openapigo migrate --from v0.5 --to v0.6 ./api/
   ```
   The `migrate` command rewrites generated code headers and adjusts import paths. For changes that require full regeneration, it simply instructs the user to regenerate.

## Consequences

### Positive

- **Single module eliminates version matrix**: Users cannot accidentally have mismatched generator/runtime versions (same `go.mod` entry)
- **Compile-time compatibility check**: `RuntimeCompatV0_X` constants catch mismatches at build time, not runtime
- **Clear pre-1.0 expectations**: Users know that minor versions may break and regeneration is required
- **Post-1.0 forward compatibility**: Like protobuf, older generated code works with newer runtimes

### Negative

- **Regeneration on every minor upgrade**: Pre-1.0, users must regenerate when upgrading. This is friction, but it's standard for code generators.
- **Single module means CLI bloat in `go.mod`**: Users who only want the runtime also pull in the CLI's dependencies. Mitigation: keep CLI dependencies minimal, or consider splitting the module post-1.0.

### Risks

- **Version constant proliferation**: Each minor version adds a `RuntimeCompatV0_X` constant. These accumulate but are trivially small (bool constants). Post-1.0, only one per major version is needed.
- **`go get` ordering**: If users run `go get` before `openapigo generate`, they get the warning about version mismatch. This is informational and self-resolving (regeneration fixes it).
