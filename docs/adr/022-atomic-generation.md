# ADR-022: Atomic Generation and Failure Recovery

## Status

Accepted

## Date

2026-03-02

## Context

Code generation can fail mid-way for many reasons:

- Invalid spec (detected late during generation, not at parse time)
- Disk full
- Permission errors
- Template rendering bugs
- Signal interruption (Ctrl+C)

ADR-019 describes stale file cleanup: before writing, the generator deletes old generated files that are no longer in the output set. If generation fails **after** the cleanup but **before** all new files are written, the output directory is left in a half-broken state — old files deleted, new files incomplete or missing. The user's project won't compile, and they must regenerate from scratch (or restore from version control).

This is a real operational hazard, especially in CI/CD pipelines where a transient failure could break the build.

### Existing tools' approaches

- **protoc**: Writes output files directly. No atomic generation. Failures leave partial output.
- **sqlc**: Writes output files directly. Provides `sqlc diff` to detect drift and `sqlc vet` to validate before generation.
- **buf generate**: Writes to a managed output directory. No explicit atomicity.
- **Terraform**: Uses state files and plan/apply separation. Different problem domain.

## Decision

### Tmp-then-Rename Strategy

The generator writes all output to a **temporary directory** first, then atomically replaces the target directory's contents on success.

#### Generation Pipeline

```
1. Parse spec
2. Resolve $ref (ADR-020)
3. Analyze schemas + operations
4. Generate all Go files in memory
5. Format generated code (gofmt)
6. Write all files to temp directory (same filesystem as output — see Risks)
7. Verify: all files written, no I/O errors
8. Atomic swap:
   a. Scan target directory for generated files (files with openapigo header)
   b. Delete those generated files from target directory
   c. Move new files from temp directory to target directory
   d. Remove temp directory
```

**Why not full directory rename**: The target directory may contain user-owned files (e.g., `doc.go`, custom extension files). We cannot replace the entire directory — we must selectively replace only generated files.

#### Failure Scenarios

| Failure point | State | Recovery |
|---|---|---|
| Steps 1-5 (parse, analyze, generate, format) | No filesystem changes yet | Clean failure. Target directory untouched. |
| Step 6 (write to temp) | Temp directory has partial files | Clean failure. Temp directory is cleaned up. Target directory untouched. |
| Step 7 (verify) | Temp directory complete but verification failed | Clean failure. Temp directory is cleaned up. Target directory untouched. |
| Step 8a-8b (scan + delete old files) | Old generated files deleted, new files not yet moved | **Interrupted state.** See below. |
| Step 8c (move new files) | Partial move | **Interrupted state.** See below. |
| Step 8d (cleanup temp) | New files in place, temp directory remains | Harmless. OS will clean up temp directory. Next run succeeds. |

#### Handling Interrupted State (Steps 8a-8c)

If the process is killed during the swap phase (steps 8a-8c), the output directory may be in an inconsistent state. To handle this:

1. **Before step 8a**, the generator writes a **manifest file** to the temp directory: `.openapigo-pending.json`
2. The manifest lists all files to be written and their checksums
3. On the **next run**, if the generator detects `.openapigo-pending.json` in the output directory OR the temp directory, it:
   - Logs a warning: `WARN: Detected incomplete previous generation. Cleaning up before proceeding.`
   - Cleans up any leftover state
   - Proceeds with fresh generation

```json
// .openapigo-pending.json (written to temp dir, moved to target at step 8a)
{
  "generator_version": "0.5.0",
  "timestamp": "2026-03-02T10:30:00Z",
  "temp_dir": "/tmp/openapigo-gen-abc123/",
  "files": [
    {"path": "types.go", "checksum": "sha256:abc..."},
    {"path": "operations.go", "checksum": "sha256:def..."},
    {"path": "endpoints.go", "checksum": "sha256:ghi..."}
  ]
}
```

The manifest file itself is deleted as the last step of a successful generation.

#### Signal Handling

The generator installs signal handlers for `SIGINT` and `SIGTERM`:

- **During steps 1-7**: Clean up temp directory and exit
- **During step 8**: Attempt to complete the swap (the swap is designed to be fast — just file moves, not writes). If a second signal is received, abandon and exit (the next run will detect and recover via the manifest)

### Interaction with Stale File Cleanup (ADR-019)

ADR-019's stale file cleanup is subsumed by the atomic generation strategy:

1. Generated files in the target directory are identified by the `// Code generated by openapigo. DO NOT EDIT.` header (same as ADR-019)
2. During the swap phase (step 8b), ALL old generated files are removed
3. New generated files are moved in (step 8c)
4. The net effect is identical to ADR-019's cleanup + write, but atomic

The `Removed stale: ...` message from ADR-019 is still emitted when old files are deleted that are not in the new output set.

### `--dry-run` Mode

For safety, users can preview what the generator would do:

```bash
openapigo generate -i petstore.yaml -o ./api --dry-run
```

Output:
```
Would generate:
  api/types.go (new)
  api/operations.go (new)
  api/endpoints.go (updated)
  api/auth.go (unchanged)
Would remove:
  api/types_legacy.go (stale)
```

No files are created, modified, or deleted.

### `--force` for Skip-Verification

By default, the generator verifies that the output directory is in a clean state (no unstaged changes to generated files, per `git status`). If generated files have local modifications:

```
ERROR: Generated file api/types.go has local modifications.
       Regeneration would overwrite them. Use --force to proceed.
```

This protects against accidentally losing manual patches. `--force` bypasses this check.

**Git detection**: This check only applies when the output directory is inside a git repository. The generator runs `git status --porcelain -- <file>` for each generated file. If git is not available or the directory is not a git repo, the check is skipped.

## Consequences

### Positive

- **No half-broken output**: If generation fails at any point before the swap phase, the target directory is untouched
- **Recoverable interrupted state**: The manifest file enables detection and cleanup on the next run
- **User files are never touched**: Only files with the openapigo header are replaced
- **Dry-run enables safe preview**: Users can see what would change before committing
- **Local modification protection**: Users won't accidentally lose manual patches

### Negative

- **Slightly more complex than direct write**: The temp directory + swap adds implementation complexity
- **Disk space**: During generation, both the temp directory and target directory exist simultaneously. For very large APIs, this doubles the disk usage briefly.
- **Step 8 is not perfectly atomic**: File moves are individually atomic (on most filesystems), but the sequence of moves is not transactional. The manifest-based recovery handles this.

### Risks

- **Cross-filesystem temp directory**: Go's `os.Rename` does NOT fall back to copy+delete on cross-filesystem boundaries — it returns an `EXDEV` (cross-device link) error. Therefore, the generator MUST create the temp directory on the **same filesystem** as the output directory. The generator creates the temp directory in the **same parent directory** as the output (e.g., `./api/.openapigo-tmp-XXXXXX/`) to guarantee same-filesystem `os.Rename`. The generator NEVER uses `$TMPDIR` for the temp directory because `$TMPDIR` is often on a different filesystem (e.g., `/tmp` on Linux is frequently a tmpfs mount). If the parent directory is not writable, the generator attempts to create the temp directory **inside** the output directory itself (e.g., `./api/.openapigo-tmp-XXXXXX/`). If that also fails (e.g., output directory does not exist yet), the generator creates the output directory first, then creates the temp directory inside it. If creating a same-filesystem temp directory fails for any reason, the generator emits a **fatal error** and exits: `ERROR: Cannot create temp directory on the same filesystem as output directory. Check directory permissions.` The generator does NOT fall back to non-atomic writes — the atomicity guarantee is unconditional.
- **Permissions**: The temp directory inherits the parent directory's permissions. If the output directory has restricted permissions, the temp directory should too.
